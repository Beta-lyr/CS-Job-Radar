## 1. 文档信息

**项目名称**：计算机学生职业方向雷达
**英文名**：CS Job Radar
**文档类型**：Technical Design Document
**版本**：V1.0 MVP
**对应 PRD**：计算机学生职业方向雷达 PRD V1.0
**目标**：实现一个低成本、可上线、可持续更新岗位数据的技术岗位趋势分析网站。

---

# 2. 技术目标

## 2.1 MVP 技术目标

V1.0 阶段的技术目标不是构建完整招聘平台，而是跑通以下闭环：

```text
公开岗位数据源
  ↓
定时采集
  ↓
原始数据入库
  ↓
岗位清洗与标准化
  ↓
方向分类
  ↓
技能抽取
  ↓
薪资与应届友好度分析
  ↓
每日/每周统计
  ↓
Next.js 网站展示
```

## 2.2 核心技术要求

1. 支持维护多个公开数据源。
2. 支持每日定时采集岗位数据。
3. 支持岗位去重、清洗、标准化。
4. 支持岗位方向分类。
5. 支持技能标签抽取。
6. 支持薪资标准化。
7. 支持应届友好度判断。
8. 支持每日方向统计。
9. 支持每周报告生成。
10. 支持低成本线上部署。
11. 支持后续平滑扩展到独立后端、用户系统、会员系统。

# 3. 技术选型

## 3.1 总体技术栈

```text
前端：
Next.js + TypeScript + Tailwind CSS + ECharts/Recharts

数据采集：
Python + Requests + BeautifulSoup + Playwright

数据处理：
Python + pandas + Pydantic + SQLAlchemy

数据库：
PostgreSQL

部署：
Cloudflare Pages / Vercel
Neon PostgreSQL / Supabase PostgreSQL
GitHub Actions 定时任务

后续可扩展：
Go API Server
Redis
Meilisearch / Elasticsearch
对象存储
```

## 3.2 技术选型理由

### Next.js

用于网站展示、SEO、报告页生成、轻量 API。

优势：

* 适合内容型网站；
* 支持静态生成和服务端渲染；
* 适合做 SEO；
* 与 Tailwind、图表库集成方便；
* 后期可以接用户系统和付费系统。

### Python

用于采集和数据分析。

优势：

* 爬虫生态成熟；
* HTML 解析方便；
* 文本处理能力强；
* pandas 适合统计分析；
* 后续接 LLM 做岗位摘要和技能抽取更方便。

### PostgreSQL

用于结构化存储和统计分析。

优势：

* 适合岗位、技能、统计数据；
* 支持 JSONB；
* 支持全文检索；
* 支持聚合查询；
* 后期可以承载大部分 MVP 分析需求。

### GitHub Actions

用于低成本定时任务。

优势：

* 免费额度够 MVP；
* 不需要维护服务器；
* 适合每日/每周低频数据任务；
* 可以直接连接 PostgreSQL 写入数据。

# 6. 数据流设计

## 6.1 数据处理主流程

```text
1. 读取启用的数据源
2. 根据数据源类型选择 Fetcher
3. 请求列表页
4. 解析岗位链接
5. 请求岗位详情页
6. 解析原始岗位字段
7. 计算 source_url_hash 和 raw_hash
8. 写入 raw_jobs
9. 对 raw_jobs 做标准化
10. 写入 jobs
11. 抽取技能写入 job_skills
12. 计算每日统计写入 daily_direction_stats
13. 每周生成报告写入 weekly_reports
```

## 6.2 采集流程

```text
Source Registry
  ↓
Fetcher
  ↓
HTML / JSON / RSS
  ↓
Parser
  ↓
RawJobDTO
  ↓
Deduplicate by source_url_hash
  ↓
Insert raw_jobs
  ↓
Crawl Log
```

## 6.3 分析流程

```text
raw_jobs
  ↓
salary_normalizer
  ↓
education_normalizer
  ↓
experience_normalizer
  ↓
direction_classifier
  ↓
fresh_graduate_classifier
  ↓
skill_extractor
  ↓
jobs + job_skills
```

## 6.4 统计流程

```text
jobs + job_skills
  ↓
按日期 / 城市 / 方向聚合
  ↓
薪资分位数计算
  ↓
技能 Top N 计算
  ↓
应届友好度计算
  ↓
daily_direction_stats
  ↓
weekly_reports
```

```text
source_url_hash = sha256(normalized_url)
```

如果 source_url_hash 已存在，则跳过。

### 第二层：内容去重

```text
raw_hash = sha256(title + company + city + description)
```

如果 raw_hash 重复，则认为是重复岗位。

### 第三层：相似度去重，后续版本

同公司、同城市、同岗位标题，JD 相似度超过阈值，则合并。

V1.0 可先不做复杂相似度去重。

# 31. 技术风险

## 31.1 数据源不稳定

风险：

```text
页面结构变化导致解析失败。
```

解决：

```text
1. 采集日志监控
2. parser_type 分层
3. 数据源失败告警
4. 每周人工检查高价值数据源
```

## 31.2 数据质量不足

风险：

```text
岗位少、重复多、分类不准，导致分析没有参考价值。
```

解决：

```text
1. 多源采集
2. 可信度评分
3. 抽样检查
4. 技能词典持续维护
```

## 31.3 合规风险

风险：

```text
采集商业平台数据可能触发规则和合规问题。
```

解决：

```text
1. 优先官方、高校、企业官网
2. 不绕登录和验证码
3. 只展示聚合结果
4. 提供数据说明和删除机制
```

## 31.4 成本增长

风险：

```text
Playwright、数据库、访问量增加导致成本上升。
```

解决：

```text
1. 前期低频采集
2. 优先静态页面
3. 使用预聚合统计表
4. 数据量上来后再拆服务
```

# 32. 最终技术方案总结

V1.0 推荐方案：

```text
Next.js + TypeScript + Tailwind + ECharts
Python Crawler + Analyzer
PostgreSQL
GitHub Actions 定时任务
Cloudflare Pages / Vercel 部署
Neon / Supabase 托管数据库
```

核心原则：

```text
1. 不做大而全招聘平台
2. 不追求实时全量数据
3. 优先跑通数据分析闭环
4. 优先输出学生能看懂的方向建议
5. 前期低成本、低复杂度部署
6. 后期根据用户和数据量逐步拆服务
```

MVP 的关键闭环：

```text
公开岗位数据
  ↓
稳定采集
  ↓
清洗标准化
  ↓
方向和技能分析
  ↓
趋势报告
  ↓
学生访问和反馈
  ↓
验证商业化
```
